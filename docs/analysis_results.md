# gstackのGoogle Antigravity対応に関する分析結果

## 概要
`gstack` は、元々 Claude Code や Codex といったCLIベースのAIエージェントを、「役割特化型の仮想エンジニアリングチーム」に進化させるために開発されたフレームワークです。
大きく以下の2つの要素から構成されています：

1. **ワークフロー（SKILL.md）の集合**: CEO、QAリード、リリースエンジニアなどの役割を持つプロンプト群。
2. **独自のブラウザ操作デーモン (`browse`)**: 高速・持続的にブラウザを操作するためのBunベースのローカルサーバー。

## Antigravity対応に向けた主要課題

Google Antigravityには強力なツール（`browser_subagent` や `run_command`）がビルトインされているため、`gstack` の機能を利用するためには以下の対応が必要です。

### 1. ワークフロー定義（ファイル構成）の変換 【今回着手】
* **現状:** 各スキルは `[skill-name]/SKILL.md` という構造で、固有のYAMLフロントマターを持っています。
* **対応策:** Antigravityのワークフロー仕様に合わせ、`.agents/workflows/[skill-name].md` というファイル構成にし、YAMLフロントマターを `description: [short title]` に変換する必要があります。

### 2. シェルコマンドの自動実行（Turboアノテーション）の追加
* **現状:** 各SKILLには、状態管理や分析用データを保存するための長大なBashスクリプト（Preamble等）が含まれています。
* **対応策:** Antigravityのデフォルトではシェル実行時にユーザー承認を求めるため、自動実行できるようにスクリプトブロックの上に `// turbo`、またはファイル全体に `// turbo-all` を付与する必要があります。

### 3. ブラウザ操作の統合手法の決定
* **現状:** 独自の `$B` コマンドでHeadless Chromeを操作します。
* **対応策:** 操作ごとにユーザー承認が必要になるターミナルベースの `$B` ではなく、自律型の `browser_subagent` に処理を委譲するように元のプロンプト（テスト手順など）を改修することが推奨されます。

### 4. `setup` スクリプトの拡張
* **対応策:** 現在の `--host (claude|codex|factory)` オプションに加え、`--host antigravity` を追加し、適切なディレクトリ(`.agents/workflows/`)にファイルを生成・リンクする処理を追加します。

### 5. Windows特有の開発環境の確認
* **現状:** BunのWindows環境特有のバグ（通信パイプ）に対応するため、`gstack` はNode.jsへのフォールバックを実装しています。
* **対応策:** Antigravity環境下（Windows）で利用する場合は、BunとNode.jsの両方がインストールされていることを前提とした処理や案内を追加する必要があります。
