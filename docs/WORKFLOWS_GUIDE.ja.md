# GStack Antigravity ワークフロー完全ガイド

GStack は単なる「AIへのコマンド集」ではなく、**「シニアエンジニア（あるいはスタートアップの創業者）のような思考と判断基準を持ったAIチームメンバー」** として振る舞うように設計されています。

そのため、初めて使う方は、AIから返ってくる「独特な口調」や「選択肢の提示方法」に少し驚かれる（あるいは混乱される）かもしれません。
このガイドでは、AIがなぜそのような挙動をするのか（GStackの基本思想）と、代表的なスラッシュコマンド（ワークフロー）の詳しい使い方を解説します。

---

## 1. なぜAIは独特な形式で質問してくるのか？

GStackを利用していると、AIから突然以下のような**フォーマット化された質問**が飛んでくることがあります。

> **AskUserQuestion**
> 1. **Re-ground**: ◯◯プロジェクトにて、初回起動に伴う確認を行っています。
> 2. **Simplify**: 〜〜〜という機能を追加しますか？
> 3. **Recommend**: RECOMMENDATION: Choose A because ... Completeness: 10/10
> 4. **Options**: 
>    A) はい (human: ~5 min / CC: ~1 min) 
>    B) いいえ (human: ~0 min / CC: ~0 min)

これはバグやエラーではありません。**GStack のコア設計である「AskUserQuestion」フォーマット**です。

### 📌 意図と意味
ソフトウェア開発において、「AIが勝手に推測して間違った大きな変更を加える」ことは非常に危険です。一方で「専門用語だらけでユーザーに判断を丸投げする」ことも非効率です。
そこでGStackは、重要な分岐点に差し掛かった際、**「絶対にこのフォーマットでユーザーに承認（または選択）を求める」** ようAIに強力な制約を課しています。

- **Re-ground (現状認識):** ユーザーが別の作業から戻ってきても「いま何の話をしているか」がすぐ分かるように、現在の状況を1〜2文で再確認します。
- **Simplify (平易な説明):** 「関数の引数が〜」といった内部コードの話ではなく、「ユーザーの目にどう映るか」「どんなメリットがあるか」を非エンジニアでも分かる言葉で説明します。
- **Recommend (AIからの推奨):** AIが「どの選択肢が最も優れていると考えるか」とその理由を提示します。
- **Completeness (網羅性スコア):** GStackには **「Boil the Lake（妥協せずやり切る）」** という強い思想があります。人間が手作業で行えば数日かかる網羅的なテストや実装も、AIなら数分で終わります。そのため、AIは常に「近道（スコア3/10）」よりも「完全な実装（スコア10/10）」を推奨するように設計されています。
- **時間コストの提示:** その作業に人間が取り組んだ場合の時間（human）と、AIが取り組んだ場合の時間（CC: Claude Code/Antigravity）を比較提示します。

**💡 ユーザー側の対応:** あまり難しく考える必要はありません。内容を読み、**「Aでお願い」** または **「Bで行こう」** とチャットで返信するだけで、AIが自律的に作業を再開します。

## 2. 主要ワークフローの詳細な使い方

ここでは、特に強力でよく使われるワークフローの「裏側で何が行われているのか」を解説します。

### `/learn` （プロジェクトの学習管理）
あなたのプロジェクト固有の「設計のクセ」「よくあるバグ」「好みのコーディングスタイル」をAIが学習し、管理するためのコマンドです。

- **何をするのか？:** 
  開発を進める中でAIが「このプロジェクトではパスカルケースを使うんだな」「このAPIはレート制限に引っかかりやすいな」といった**学び（Learnings）**を発見すると、自動的にJSONデータとして蓄積していきます。`/learn` はその蓄積された学習履歴を引き出し、整理・確認するためのツールです。
- **裏側の挙動:** 
  現在のプロジェクトリポジトリ内の `.gstack` フォルダ内にある `learnings.jsonl` を解析し、最新の20件を表示します。間違った学習データがあれば、ここで削除（Prune）するようAIに指示することも可能です。

### `/autoplan` （多角的な自動レビュー・パイプライン）
コードを書き始める「前」に、AIが3つの異なる視点で実装方針をボコボコに（しかし建設的に）レビューする機能です。

- **何をするのか？:** 
  新機能を作りたい時、いきなりコードを書かせるのではなく、まず Markdown に「やりたいこと（プラン）」を書きます。その後 `/autoplan` を実行すると、AIが内部で以下の3つの人格（サブルーチン）を順番に呼び出します。
  1. **CEO Review (`/plan-ceo-review`)**: 「そもそもこの機能はユーザーにとって10つ星の体験か？」「無駄に複雑すぎないか？」をビジネス・プロダクト目線で厳しく問います。
  2. **Design Review (`/plan-design-review`)**: 「UIに統一感はあるか？」「アニメーションは適切か？」をデザイナー視点でチェックします。
  3. **Eng Review (`/plan-eng-review`)**: 「アーキテクチャの依存関係はおかしくないか？」「テスト可能な状態か？」をシニアエンジニア視点でチェックします。
- **裏側の挙動:** 
  これら各レビュー機能がプランファイル（例: `implementation_plan.md`）を読み込み、それぞれの視点から指摘事項や修正案をプランに直接書き込み、追記していきます。最後にすべてを取りまとめた「レビューレポート」を出力して完了します。

### `/review` （PR・差分の自律レビュー）
あなたが書いた（またはAIに書かせた）コードが、安全で本番環境に耐えうるかを確認するコマンドです。

- **何をするのか？:** 
  まるでシリコンバレーの厳しいシニアエンジニアのように、現在のコミット差分や未コミットの変更をスキャンします。特に**「SQLインジェクションの脆弱性はないか」「AIに渡すプロンプトとユーザー入力の境界線（トラストバウンダリ）は守られているか」「非同期処理の競合はないか」**といった、重大なバグ・セキュリティインシデントに直結する項目を専門的に検証します。
- **裏側の挙動:** 
  `git diff` やファイル変更履歴を読み取り、GStack のプロンプト内に定義された厳格なチェックリストに照らし合わせてコードを解析し、問題があればレポートと修正のアドバイスを出力します。

### `/qa` （自律型ブラウザ品質保証テスト）
ブラウザを自動で立ち上げ、ウェブアプリの動作テストやUIの崩れをAI自身の目で確認させるコマンドです。

- **何をするのか？:** 
  例えば「`/qa http://localhost:3000 にアクセスして、ログイン機能からダッシュボードの表示までをテストして`」と指示するだけで、AIが背後でブラウザを起動（browser_subagent）します。画面のスクリーンショットやDOM（HTML構造）を読み取りながら、自律的にクリックやテキスト入力を繰り返し、エラーログが出ていないか、ボタンが隠れていないかをチェックします。
- **裏側の挙動:** 
  GStack オリジナルのレガシーなコマンド（`$B`）を、Antigravity ネイティブのブラウザ操作ツールに自動翻訳して委譲しています。もしUIの崩れやJavaScriptエラーを発見した場合、AIはそれを報告するだけでなく、**その場でコードを編集してバグを修正する**ところまでを一貫して試みます。（※テストレポートだけが欲しい場合は `/qa-only` を利用します）

## 4. 各ワークフローの前提条件（依存関係）

GStack のワークフローは、プロジェクト内の特定のファイルや状態を前提として動作します。実行前にこれらが揃っているか確認してください。

| コマンド | 必要なファイル・状態 | 補足説明 |
| :--- | :--- | :--- |
| **`/autoplan`** | `implementation_plan.md` | 実装の「設計図」が必要です。ない場合は会話から生成されます。 |
| **`/plan-ceo-review`** | `implementation_plan.md` | CEO視点でのレビュー対象となるプランファイルが必要です。 |
| **`/review`** | Git の差分 (diff) | `main` ブランチ等との差分をレビューします。コミット前でもOKです。 |
| **`/qa`** | 動作中のURL | ローカルまたはリモートのURLが必要です（自動検知も試みます）。 |
| **`/ship`** | 全テストの通過 | コードがデプロイ可能な状態であることを前提とします。 |
| **`/learn`** | `.gstack/learnings.jsonl` | 過去の学習データ。自動的に生成・更新されます。 |

### 📌 重要なディレクトリ構造
- **`.agents/workflows/`**: このディレクトリ内の Markdown ファイルが、Antigravity 上で実行される「脳」の実体です。GStack 自体のアップデートにより、ここが常に最新に保たれます。
- **`GEMINI.md`**: AI がリクエストを適切なワークフローにルーティングするための「交通整理」を行います。

---

> 💡 **Tip:** 万が一ワークフローが正常に動作しない場合は、`gstack` サブモジュールの状態や、`.gstack-preamble.sh` が正しく実行されているかを確認してください。

---

> 💡 **Tip:** GStackのすべてのワークフローは、各プロジェクトの `.agents/workflows/` フォルダに Markdown 形式で保存されています。これらはただの設定ファイルではなく、**AIに対する精密なプロンプト（指示書）**そのものです。時間がある時にぜひ中身を眺めてみてください。世界トップクラスの「AIを上手く操るためのプロンプトエンジニアリング」の真髄を学ぶことができます。
